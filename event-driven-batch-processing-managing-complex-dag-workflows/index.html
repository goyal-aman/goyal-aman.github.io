Event-Driven Batch Processing: Managing Complex DAG Workflows

28 Feb, 2026

Core Concept

  • Extend work queues into workflows (DAGs) where the output of one stage is the input to the next
  • Pub/Sub (Kafka, SQS) serves as the “glue” that provides reliable message delivery and decoupling

Core Patterns

  • Copier: Duplicates work into multiple queues (e.g., rendering video into 4K and GIF simultaneously)
  • Filter: Removes unwanted items via an adapter over the source
  • Splitter: Routes items to different queues based on conditions (e.g., Email vs. SMS)
  • Sharder: Distributes work evenly for fault isolation and scaling
  • Merger: Combines multiple queues into one

Real-World Problems and Solutions

  • Stragglers: Use work stealing where idle workers take from long queues
  • Poison Messages: Tasks that always fail should be handled with retry counts and eventual drops to avoid crashing workers
  • Backlogs: Use priority queues to ensure new work is not delayed by retried work

One-line Summary

Event-driven batch systems connect multiple work queues into workflows using patterns like copier and splitter to enable scalable data processing pipelines.

Understanding the trade-offs between Join for completeness and Reduce for performance.” tags = [‘Distributed Systems’, ‘Batch Processing’, ‘MapReduce’, ‘Synchronization’] +++

Core Concept

  • Focus on bringing parallel work back together for aggregation or final results

Core Patterns

1. Join (Barrier Synchronization)

  • Waits until ALL parallel work is done before releasing results
  • Ensures completeness and correctness, but is slower due to waiting

2. Reduce

  • Incrementally combines outputs and can start before all data is ready
  • Highly parallel and faster than Join; includes special cases like Sum and Histogram

Tree-like Reduction

  • Reduce is composable; you can run partial reduces in parallel to form a “tree” that leads to the final result

Trade-offs

  • Join: Guaranteed correctness but high latency
  • Reduce: Fast and parallel but requires more complex logic to implement

One-line Summary

Coordinated batch processing introduces join and reduce patterns to safely combine parallel work into final results.

+++ title = “Monitoring and Observability: The Four Pillars of System Health” date = “2026-03-14” description = “A deep dive into Logging, Metrics, Alerting, and Tracing to build distributed systems that are debuggable and reliable.” tags = [‘Distributed Systems’, ‘Observability’, ‘SRE’, ‘Monitoring’] +++

Core Concept

  • Failures in distributed systems are inevitable; observability allows you to debug from the outside and detect problems before users do

The Four Pillars

  1. Logging: Records individual events with request IDs; focus on errors and unusual behavior
  2. Metrics: Time-series aggregate views (counts, rates, histograms) of request/error rates and latency
  3. Alerting: Notifications based on your SLO definitions; must be actionable to avoid alert fatigue
  4. Tracing: Tracks a single request across multiple services using a correlation ID to find bottlenecks

Storage and Management

  • Use log aggregation to combine logs across services
  • Implement downsampling and tiered storage to manage the massive volume of data produced by distributed systems

One-line Summary

Monitoring and observability combine logs, metrics, alerting, and tracing to detect and debug issues before users are impacted.

+++ title = “AI Inference and Serving: Patterns for Production Intelligence” date = “2026-03-21” description = “Deploying AI models at scale. Exploring prompt engineering, RAG, and the probabilistic nature of testing AI systems.” tags = [‘Distributed Systems’, ‘AI’, ‘Inference’, ‘RAG’] +++

Core Concept

  • Inference is using a learned function (model) in production, typically exposed via a REST API

Retrieval-Augmented Generation (RAG)

  • Solves the problem of models being static or generic
  • Combines a user query with external data from a database or API before sending it to the model for up-to-date, personalized responses

Deployment and Distribution

  • Use standard formats like ONNX and model hubs like Hugging Face
  • Cache models locally to avoid repeated downloads
  • Use canary deployments to roll out models gradually

Testing AI Systems

  • Traditional exact-match testing doesn’t work; AI testing is probabilistic
  • Measure overall quality and statistical performance across many inputs; you can even use AI to evaluate AI responses

One-line Summary

AI inference systems serve models via APIs, where prompt engineering and RAG are key to delivering accurate, scalable applications.
